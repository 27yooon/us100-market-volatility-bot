# 쭈꾸미 앱 전환 계획

이 문서는 Render 모의매매 검증 결과가 충분히 좋아졌을 때, 쭈꾸미를 앱/웹 대시보드로 전환하기 위한 기준과 기능을 정리한다.

현재 단계는 실거래가 아니라 모의매매 검증이다. 승률 80% 이상이 나오더라도 표본 수, 손익비, 손실 구간, 전략 버전별 성과를 함께 확인하기 전까지 실거래 앱으로 전환하지 않는다.

## 앱 전환 조건

앱 전환 검토는 아래 조건을 최소 기준으로 한다.

| 기준 | 최소 조건 |
|---|---|
| 실제 모의매매 표본 | 전략별 최소 30회 이상 |
| 후보/미진입 표본 | 전략별 최소 50건 이상 |
| 검증 기간 | 최소 2주 이상, 가능하면 4주 이상 |
| 승률 | 80% 이상이면 앱 전환 검토 가능 |
| 손익 | 총 손익 양수, Profit Factor 1.5 이상 우선 |
| 위험 | Max Drawdown과 연속 손실이 감당 가능한 범위 |
| 분리 | 전략 버전별로 결과가 섞이지 않아야 함 |

## 현재 전략 구분

| 이름 | 코드 이름 | 앱에서의 의미 | 현재 상태 |
|---|---|---|---|
| 쭈꾸미 원본 | `zukkumi_original` | 우리가 만든 P라인/라운딩/피봇/EMA 기준 | 실제 모의매매 |
| 기본지표 | `indicator_basic` | Bollinger/EMA/RSI/ATR 기본 비교군 | 실제 모의매매 |
| 오픈박스 매매 | `orb_paper` | NY 오픈 22:30~23:00 KST ORB/Initial Balance | 실제 모의매매 |
| 점수제 관찰 | `score_watch` | EMA/RSI/MACD/Stochastic/ATR 점수 후보 | 관찰 전용 |

## 앱 핵심 화면

### 1. 운영 대시보드

첫 화면은 현재 감시 상태를 바로 보여준다.

필수 표시:
- Render 실행 상태
- 현재 감시 종목 `NQ=F`
- 현재 가격
- 전략별 켜짐/꺼짐
- 현재 열린 모의 포지션
- 오늘 신호 수
- 오늘 손익
- 이번 주 손익
- 마지막 heartbeat 시각
- 데이터 오류/서버 오류 여부

### 2. 전략별 화면

전략마다 같은 기준으로 성과를 비교한다.

필수 표시:
- 전략명
- 전략 버전
- 현재 상태: 실제 모의매매 / 관찰 / 중지
- 진입 조건
- 무효 조건
- 최근 10개 거래
- 승률
- 손익합
- Profit Factor
- Max Drawdown
- 연속 손실
- 롱/숏별 성과
- 시간대별 성과

### 3. 거래 복기 화면

각 거래를 사람이 다시 볼 수 있게 만든다.

필수 표시:
- 진입 시각
- 청산 시각
- 방향
- 진입가
- 손절가
- 익절가
- 결과
- 손익
- 진입 근거
- 주의 사항
- 무효 기준
- 차트 이미지 또는 차트 재생성 링크
- 복기 메모
- 평가: 좋은 거래 / 운 좋게 간 거래 / 안 들어갔어야 한 거래

### 4. 후보/미진입 화면

자동매매로 발전하려면 들어간 자리만큼 안 들어간 자리도 중요하다.

필수 구분:
- 실제 진입
- 놓친 진입
- 안 들어간 게 맞았던 후보
- 애매한 후보
- 관찰 전용 후보

필수 표시:
- 후보 발생 시각
- 전략명
- 방향
- setup
- 후보 가격
- 무효 기준
- 후행 결과
- `MISSED_ENTRY`, `FILTERED_OK`, `AMBIGUOUS`
- 왜 진입하지 않았는지

### 5. 리스크 관리 화면

승률이 높아도 안전장치가 없으면 앱으로 만들면 안 된다.

필수 안전장치:
- 하루 최대 손실 제한
- 주간 최대 손실 제한
- 연속 손실 정지
- 장 시작 직후 과변동 정지
- 지표 발표 전후 진입 금지
- 데이터 끊김 시 진입 금지
- Render heartbeat 지연 시 진입 금지
- 모의 모드와 실거래 모드 분리

### 6. 알림 화면

텔레그램은 상태 알림 중심으로 사용한다.

필수 알림:
- Render 시작/중단
- heartbeat 지연
- 실제 모의매매 OPEN/CLOSE
- 하루 손실 제한 접근
- 전략별 이상 상황
- Notion 기록 실패

## 데이터 구조

앱 전환을 위해 아래 테이블이 필요하다.

| 테이블 | 역할 |
|---|---|
| `strategies` | 전략 이름, 코드 이름, 현재 상태 |
| `strategy_versions` | 룰 변경 이력, 적용 시작/종료 시각 |
| `signals` | 모든 신호와 후보 |
| `paper_trades` | 실제 모의매매 진입/청산 |
| `missed_candidates` | 미진입 후보와 후행 결과 |
| `daily_summaries` | 일별 전략별 요약 |
| `risk_events` | 손실 제한, 중단, 오류 |
| `server_heartbeats` | Render 실행 상태 |
| `reviews` | 사람이 남긴 복기 메모 |

## 백테스트/리포트 필수 지표

승률만 보면 안 된다. 앱에는 아래 지표가 있어야 한다.

- 거래 수
- 승률
- 총 손익
- 평균 손익
- Profit Factor
- Max Drawdown
- 연속 손실
- 롱 승률 / 숏 승률
- 시간대별 승률
- 요일별 승률
- 전략 버전별 성과
- 미진입 후보의 후행 성과

## 단계별 전환

| 단계 | 목표 |
|---|---|
| 1단계 | Render 모의매매 안정화 |
| 2단계 | 자동 리포트 생성 |
| 3단계 | 웹 대시보드 MVP |
| 4단계 | 모바일 대응 |
| 5단계 | 알림 앱처럼 사용 |
| 6단계 | 소액 실거래 검토 |
| 7단계 | 완전 자동화 검토 |

## 앱 MVP 우선순위

처음 앱은 화려하면 안 된다. 반복해서 보는 운영 도구여야 한다.

1. 운영 대시보드
2. 전략별 성과표
3. 거래/후보 기록표
4. 복기 메모
5. 리스크 정지 규칙
6. Render/Notion/Telegram 상태 확인

## 금지 원칙

- 승률만 보고 실거래 전환 금지
- 전략 버전 섞어서 성과 판단 금지
- 후보 기록 없이 실제 진입만 보고 판단 금지
- 데이터 오류 상태에서 진입 금지
- 사용자 확인 없는 결제, 실거래, 외부 발송 금지
